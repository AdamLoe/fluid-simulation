# v1.7-v1.10 orchestrator

## Mission

Drive the v1.7.0 through v1.10.0 plans as one sequenced effort without losing the
dependencies between UI shell work, particle dispatch legality, evidence-driven
particle performance follow-up, and water-look rendering. Done means all four named
plans have shipped, their required gates and capture/profiler evidence are recorded,
durable facts are migrated into architecture/decisions docs, and the plans are marked
`shipped + okay_to_delete: true`.

## Dependency map

- v1.7 and v1.10 both touch settings/web/render presentation; run code workers
  sequentially and verify the final UI state after both.
- v1.8 must precede v1.9 because v1.9 is intentionally evidence-gated.
- v1.8 particle dispatch and v1.10 particle rendering are separate GPU surfaces, but
  this repo has no worktree isolation, so their code edits still serialize.
- Real-GPU capture/profiler work is scarce and evidence-heavy. Per-stream captures run
  only where the active plan requires them; consolidated final capture runs once.

## Orchestrator operating rules

Use [`../agent-context/orchestrating.md`](../agent-context/orchestrating.md) and the
generic orchestration rules. Code-touching agents run sequentially in this repo.
Read-only audits may run in parallel.

- Keep this streams table current from observed agent results, not assumptions.
- Do not read broad source files in the hub context unless deciding a routing issue.
- Dispatch each stream against the active subplan and smallest required code/docs
  surface.
- Require every implementation agent to report files changed, exact gate command and
  result, deferrals, and doc updates needed.
- Do not mark a stream done until its gates and evidence are recorded here.

## Shared gates

For code changes, use the in-WSL commands from
[`../agent-context/build-run.md`](../agent-context/build-run.md):

```
cd /home/adamg/fluid-simulation/app && cargo build --target wasm32-unknown-unknown
cd /home/adamg/fluid-simulation/app && cargo test --lib
```

For visible/GPU behavior, run the static app path and `tools/capture.mjs` as documented
in build-run. Performance claims need raw profiler/capture evidence: grid dimensions,
particle count, pressure iterations, render mode, frame time, timing source, and top
costs.
