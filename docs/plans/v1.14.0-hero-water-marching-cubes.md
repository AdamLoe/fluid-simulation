# v1.14.0 — Hero water: marching-cubes surface reconstruction

> **DE-RISK OUTCOME (2026-06-09): STOP — marching cubes NOT built.** The throwaway
> occupancy-boundary-quad proxy was implemented and captured against the v1.12 screen-space
> composite on DamBreak, DoubleSplash, and a mid-flight thin tongue (valid A/B — toggle
> verified via visible voxel facets, `[harness] EVAL ->` echo present in every capture).
> The occupancy surface **clearly lost in all three pairs, worst on the thin tongue** (it
> disintegrated into stippled noise rather than a coherent blocky surface). At 64³ the
> occupancy surface is too coarse for thin features. With the baseline already ~19–24 ms/
> frame (over the 16 ms budget) and dense MC being the series' heaviest single addition, the
> gate's bar (a *clear* visible win) was not met by a wide margin. Per the plan's own
> de-risk logic, we stop and keep screen-space. **This re-affirms — does not reverse — the
> "removed surface path" decision in `decisions/rendering.md`.** The concrete plan below is
> retained as the design record should this be revisited at higher render-grid resolution.
> Throwaway quad code was reverted; the reusable capture-EVAL invocation it surfaced is
> recorded in [`hero-water-delivery-hub.md`](hero-water-delivery-hub.md).

> Hero-water series plan 3 of 7. Inherits the *Cross-cutting decisions* in
> [`v1.12.0-hero-water-refraction.md`](v1.12.0-hero-water-refraction.md). **This is the
> largest, riskiest plan in the series, and it reverses a deliberate decision.** Treat
> the de-risk experiment below as a real gate, not a formality.

## Mission

Add a reconstructed water **surface** as an alternate source for the hero material:
splat particles into a dense render-density grid, smooth it, extract an iso-surface with
marching cubes, generate normals, and feed that mesh's normal + front/back depth into the
same refraction/absorption/Fresnel composite from v1.12. Done **only if** the
reconstructed surface visibly beats the v1.12 screen-space composite on the dam-break and
thin-sheet captures; otherwise the plan exits at the experiment and we keep screen-space.

## The reversal this plan undoes — read first

[`decisions/rendering.md`](../decisions/rendering.md) and
[`architecture/rendering.md`](../architecture/rendering.md) §"Removed surface path" state
that an extracted-surface renderer (marching-cubes modules, shaders, host tables,
settings, web controls) was **deliberately removed**: "there is no extracted-surface
compatibility path." This plan brings it back. That's a legitimate change of direction,
but it must be a *decision*, not an accident — the ship-time migration **must** rewrite
those two sections, not just append to them.

## De-risk gate (do this before building the full pipeline)

1. Implement **occupancy-boundary quads only** (the source plan's Mode 1 / emergency
   fallback): generate quads on Liquid/Air cell boundaries, feed them through the v1.12
   material. ~½ day, throwaway-quality.
2. Capture Scene 3 (dam break) and Scene 5 (thin sheet) with **quads vs the v1.12
   refractive composite**.
3. **Decision point:** if the blocky-but-coherent surface already reads better than
   screen-space, invest in dense marching cubes (below). If it doesn't, stop here, delete
   the experiment, and the series skips to v1.15 with screen-space as the surface. Record
   the call in the roadmap.

This answers the open strategic question cheaply instead of spending days building MC on
faith and discovering it doesn't beat what we already shipped in v1.12.

## Scope (if the gate passes)

**In:**
- Dense render-density grid (sized from the *actual* `nx·ny·nz`, not the source plan's
  4M/2M caps — re-derive). Fixed-point particle splat consistent with the P2G
  `scatter.wgsl`/`normalize.wgsl` integer-atomics pattern.
- Density smoothing pass (1–4 iterations, tunable).
- Marching cubes over the dense grid → vertex/index (or indirect) buffers with GPU
  counters; density-gradient normals.
- `HeroSurfaceSource` Live enum: `ScreenSpace | MarchingCubes` (default stays
  `ScreenSpace`; MC is opt-in). The mesh feeds the **same** composite material.
- Optional **mesh front/back depth thickness** as an alternate thickness source (the
  source plan's thickness "track D", folded in here since it only matters with a mesh).
  Existing screen-space particle thickness stays the default.
- Debug views: `DensityGridSlice`, `ActiveSurfaceCells`, `MeshNormals`, `MeshWireframe`,
  `MeshTriangleCount`.

**Out:** SDF-field MC and active-cell compaction (follow-ups, source-plan Mode 3/active);
adaptive/sparse grids (explicitly deferred); anything that requires the mesh to be
watertight before the rest of the series works.

## Watch-outs

- **Keep screen-space as the always-available fallback.** If MC produces garbage on a
  scene, the water view must degrade to the v1.12 path, not to nothing.
- **No readback.** Vertex/index buffers + counters live on the GPU; draw via indirect.
  Triangle-count for the stats overlay comes from the indirect buffer or a *throttled*
  timing read — never a per-frame map.
- **Numbers re-derived from the tank.** 64³ render grid = 262k cells → at most a few
  hundred k triangles; the 4M-vertex caps in the source plan signal it wasn't sized to
  this `[-1,1]³` box.
- **Blobby vs blocky tuning** is the main quality risk: iso value, splat radius, smoothing
  iterations, mesh Laplacian. Expose them all Live; thin sheets are the hard case (Scene
  5) — they should not balloon into giant blobs.
- **Perf.** Splat + smooth + MC + draw on top of the v1.12 passes at ~254k particles in
  WASM/WebGPU is the series' heaviest single addition. Profile it; this is the most likely
  place to blow `target_frame_ms` and need the screen-space fallback under dynamic quality.

## Concrete implementation (orchestrator-refined, 2026-06-09)

This section is grounded in the *current* runtime on branch `hero-water-14-18`. Symbols
named below were verified against the source; the ChatGPT source plan's scene numbers and
4M caps do **not** match this tank and are corrected here.

### Ground-truth facts (verified)

- **Tank size:** `sim::H = 2.0/64.0`; default grid `64×64×64` (`grid.res_x/y/z`, range
  16–128). `GpuFluid::tank_bounds()` → `lo = -[nx,ny,nz]·H/2`, `hi = -lo`, so the default
  tank is exactly `[-1,1]³`. **262144 cells**, not 4M.
- **Cell types live on the GPU** in `GpuFluid::cell_type` (`gpu/fluid.rs`), a
  `array<u32>` of `cell_count` entries, written by `shaders/classify.wgsl`:
  **Solid=0, Liquid=1, Air=2** (boundary cells are always Solid). Accessor:
  `fluid.cell_type_buffer() -> &wgpu::Buffer` (already exposed; the `slice` renderer binds
  it). Cell linear index is `c = i + nx*(j + ny*k)`.
- **The composite material** is `composite::CompositeRenderer` driving
  `shaders/composite.wgsl` (concatenated after `shaders/env.wgsl`). It is a **full-screen
  triangle** that reads, per binding (group 0): `0` sampler, `1` thickness_tex (R16F),
  `2` whitewater_tex (R16F), `3` smoothed_z_tex (R16F — the **front eye-distance**, sentinel
  `>= 60000` = no water), `4` Params UBO, `5` Hero UBO, `6` scene_color (Rgba16Float),
  `7` scene_depth (R16F eye distance, far sentinel `65504`), `8` Cam UBO. It does **not**
  read the mesh directly — it consumes the three screen-space textures
  `thickness / whitewater / smoothed_z`. **That is the integration seam.**
- **Front-z convention:** the particle thickness pass (`shaders/particles.wgsl`
  `fs_thickness`) writes `nearest_z = clamp(center_eye_distance - radius*nz, 0, 65504)` into
  `nearest_z_view`; `WaterSmoothRenderer` (`gpu/smoothing.rs`, `water_smooth.wgsl`) blurs
  that X then Y into `smooth_z_view`, which the composite samples as binding 3. `thickness`
  is a normalized world path length; `whitewater = thickness * speed_fraction`.
- **Render path:** `gpu/mod.rs::render()` `RenderMode::Water` arm records, in order:
  (a) "hero scene prepass" → `scene_color_view` + `scene_depth_view` + `depth_view`
  (skybox, `environment.draw`, `wireframe.draw_scene`); (b) "water thickness pass" →
  `thickness/nearest_z/whitewater` (`particles.draw_thickness`); (c)+(d) smooth X/Y →
  `smooth_z`; (e) "water composite pass" → swapchain `view` (`composite.draw`);
  (f) optional diffuse. `wireframe.draw_scene` (`gpu/renderer.rs`) is the template for any
  new mesh: MRT writing color→scene_color + eye-distance→scene_depth, depth-tested against
  `depth_view`.
- **Scenes:** only **3** presets exist — `ScenePreset` in `scene/mod.rs`:
  `FallingBlob=0`, `DamBreak=1`, `DoubleSplash=2` (`scene.preset`, Reset-class,
  `enum_options` labels `["Falling blob","Dam break","Double splash"]`). **There is no
  Scene 3 / Scene 5.** The source plan's "Scene 3 dam break / Scene 5 thin sheet" must be
  reinterpreted (see scene_control below).
- **Settings/enum mechanism:** a `Setting { id, apply: ApplyClass::{Live,Reset,Reload}, … }`
  in `Registry::default()` (`settings/mod.rs`); enum labels added in `enum_options(id)`.
  Live `render.hero.*` ids are batch-applied: `lib.rs::set_setting` sees the
  `render.hero.` prefix and calls `gpu.set_hero_params(settings.hero_params())` — **no
  per-id plumbing and no rebuild**. That is the cheapest place to hang a debug toggle.

### 1. De-risk spec — throwaway occupancy-boundary-quad surface

Goal: emit one screen-facing-correct quad on every Liquid↔Air (and Liquid↔Solid-interior)
face, rasterize it through a tiny mesh pass that writes the **same three textures** the
composite already consumes, then composite unchanged. No new material, no readback.

- **New module `gpu/occupancy_surface.rs` + `shaders/occupancy_surface.wgsl`.** Bindings:
  `cell_type` storage buffer (`fluid.cell_type_buffer()`), a small UBO holding
  `view_proj` (box model already folded in by caller, same as wireframe), `nx,ny,nz`,
  `H`, tank `origin = lo`, and the camera eye-distance basis. No vertex/index buffer:
  drive it as a **non-indexed instanced draw** — one instance per *candidate boundary
  face*. Simplest correct generator without a compaction pass: dispatch
  `draw(0..6, 0..(cell_count*3))` (6 verts/quad, 3 positive-axis faces per cell), and in
  the vertex shader decode `instance/3 = c`, `instance%3 = axis`; compute the neighbour
  `c+stride[axis]`; if `cell_type[c]==Liquid XOR cell_type[neighbour]==Liquid` is false,
  emit a degenerate quad (collapse all verts to one clip point) so it's culled. This
  avoids a GPU counter for the throwaway and keeps it ~½ day.
- **What the quad writes (MRT, mirroring `fs_thickness` semantics):**
  - target 0 `thickness`: a constant slab thickness per covered pixel, e.g. `0.5*H *
    n_faces_along_view` — for the throwaway, just write `~1.5*H` (a constant) so the
    composite's Beer-Lambert/opacity has something monotonic; expose a Live scalar.
  - target 1 `nearest_z`: the fragment's **eye distance** (`clip.w` from the same
    perspective convention `particles.wgsl` uses: `center_clip.w` = `-z_eye`). Use the
    quad's front face → this becomes `smooth_z` after the existing X/Y blur, which the
    composite reads as the surface depth and finite-differences for the normal.
  - target 2 `whitewater`: 0 for the throwaway (or `thickness * const`).
  Reuse `create_r16_target` views; reuse the existing thickness render-pass attachment set
  (same formats R16F/R16F/R16F + `depth_view` load) so **no composite change is needed** —
  the quad pass simply *replaces* `particles.draw_thickness` as the producer of those three
  textures when the toggle is on (see toggle_mechanism). The existing smooth X/Y passes and
  `composite.draw()` run unchanged, giving free refraction/absorption/Fresnel/reflection on
  the blocky surface.
- **Why this is honest:** the composite's normal comes from `smoothed_z` finite differences
  (`water_normal` in composite.wgsl), so a blocky front-depth surface yields blocky-but-
  coherent normals — exactly the "blocky but coherent" read the gate is judging.
- **Throwaway discipline:** gate it behind one debug enum value; no settings-table churn
  beyond that single id; delete the module + shader + enum arm if the gate fails.

### 2. scene_control (capture harness)

Scenes are switched via the **Reset-class** `scene.preset` enum, which only takes effect
after `app.reset()` (panels.js does `set_setting` then `reset()` for `scene.preset`). The
capture harness drives this through `EVAL` against `window.__fluid` (exposed in
`web/main.js`), which `tools/capture.mjs` runs then waits `EVAL_WAIT` ms.

- **Dam break (the source plan's "Scene 3"):** preset index **1**.
  ```
  EVAL='window.__fluid.set_setting("scene.preset",1); window.__fluid.reset(); "dam"' \
  EVAL_WAIT=4000 \
  cmd.exe /c '... node capture.mjs http://localhost:5184/ dambreak-quads.png 3500'
  ```
- **Thin sheet (the source plan's "Scene 5" — DOES NOT EXIST as a preset).** Honest
  options, in preference order:
  1. Closest existing preset is **`DoubleSplash=2`** (two columns collide → thin colliding
     sheets/crowns mid-tank); use `set_setting("scene.preset",2); reset()`.
  2. For a *true* thin sheet, capture **mid-flight of DamBreak** (preset 1) a few hundred ms
     after release, when the front is a thin advancing tongue — drive with the harness
     `FRAMES`/`FRAME_INTERVAL` env (capture.mjs supports timed sequence frames) or a larger
     `EVAL_WAIT` to let the sheet thin out, then capture.
  Recommend documenting the gate as **DamBreak (front tongue) + DoubleSplash (collision
  sheet)** as the two stressors, and note in the roadmap that a dedicated thin-sheet preset
  was never added. If a real thin-sheet preset is wanted, add a 4th `ScenePreset` arm
  (e.g. a wide shallow slab `[0.05,0.05,0.05]–[0.95,0.18,0.95]`) + label — a ~5-line change,
  but that is scene work, not MC work.

### 3. toggle_mechanism (occupancy-quads vs v1.12 screen-space at capture time)

Add a single **Live debug enum** `render.hero.debug_surface_source` (values
`0=ScreenSpace`, `1=OccupancyQuads`) in the settings table. Because its id starts with
`render.hero.`, `lib.rs::set_setting` already routes it through
`gpu.set_hero_params(settings.hero_params())` with **no new plumbing** — add a field to
`HeroParams` and a `bool`/`u32` mirror the gpu side reads. In `gpu/mod.rs::render()`'s
`RenderMode::Water` arm, branch the *producer* of the thickness/nearest_z/whitewater MRT:
when the toggle is `OccupancyQuads`, replace the `particles.draw_thickness(...)` call with
`occupancy_surface.draw(...)` into the **same** "water thickness pass" attachments;
everything downstream (smooth X/Y, composite) is identical. Default stays `ScreenSpace` so
nothing ships changed. Capture-time switch:
```
EVAL='window.__fluid.set_setting("render.hero.debug_surface_source",1); "quads"'
```
This is also exactly how the full-MC `HeroSurfaceSource` enum will toggle later (Live,
no rebuild), so the throwaway's seam is the real seam.

### 4. mc_plan_summary (full MC, sizes re-derived from 262144 cells)

Build only if the gate passes. Pipeline (all GPU, no readback), sized to the **64³ tank**:

- **Render-density grid** `R16F`/fixed-point, dimensions = render-grid (start at the sim
  grid `64³ = 262144` cells; allow a `render_grid_scale` of 1–2× per axis → up to `128³ =
  2.1M` cells worst case, still small). Splat particles with the **same fixed-point integer
  atomic** contract as `scatter.wgsl`/`normalize.wgsl` (`atomicAdd(i32(round(w*scale)))`
  into num/den, then normalize), reusing `geom.w = fixed_scale`. Buffer: `cell_count *
  4 bytes` ≈ **1 MB** at 64³ (vs the source plan's multi-MB caps).
- **Smoothing:** 1–4 Jacobi/box iterations over the density grid (`smoothing_iterations`,
  `smoothing_strength`), ping-pong buffers (~1 MB each).
- **Marching cubes:** one compute thread per cell; classic 256-entry edge/tri tables as
  WGSL `const` arrays (reintroduce a **host-side table** in a pure module for
  `cargo test --lib` parity, per the exit gate). Emit triangles into a vertex buffer via a
  **GPU append counter** (`atomicAdd` on a `DrawIndirect`/counter buffer), then
  `draw_indirect`. **Vertex cap re-derivation:** at most ~5 tris/active cell; thin-shell
  active cells at 64³ are O(few×10⁴) → budget **~256k vertices** (≈ 256k·32B = 8 MB),
  generous. NOT 4M.
- **Normals:** density-gradient (central differences on the smoothed grid) at each vertex;
  optional `normal_smoothing`. Feed the mesh into the **same composite** by rasterizing it
  in the "water thickness pass" exactly like the occupancy quads: write front eye-distance →
  nearest_z, slab/front-back thickness → thickness. (Optional mesh front/back depth
  thickness as an alternate thickness source — "track D" — folded in here.)
- **`HeroSurfaceSource` Live enum** `ScreenSpace | OccupancyQuads(debug) | MarchingCubes`,
  default `ScreenSpace`; same Live `render.hero.` routing as the toggle above.
- **Fallback (load-bearing):** if MC produces a near-empty indirect count (garbage scene)
  the Water arm falls back to `particles.draw_thickness` for that frame; screen-space is
  always available (baseline is already ~24 ms @ 247.5k particles, over the 16 ms budget, so
  MC perf is a real risk and the fallback must stay).
- **Debug views** (reuse the existing `render.hero.debug_view` enum pattern in
  composite.wgsl / `enum_options`): density slice, active surface cells, mesh normals,
  wireframe, triangle count.
