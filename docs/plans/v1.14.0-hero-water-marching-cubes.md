# v1.14.0 — Hero water: marching-cubes surface reconstruction

> Hero-water series plan 3 of 7. Inherits the *Cross-cutting decisions* in
> [`v1.12.0-hero-water-refraction.md`](v1.12.0-hero-water-refraction.md). **This is the
> largest, riskiest plan in the series, and it reverses a deliberate decision.** Treat
> the de-risk experiment below as a real gate, not a formality.

## Mission

Add a reconstructed water **surface** as an alternate source for the hero material:
splat particles into a dense render-density grid, smooth it, extract an iso-surface with
marching cubes, generate normals, and feed that mesh's normal + front/back depth into the
same refraction/absorption/Fresnel composite from v1.12. Done **only if** the
reconstructed surface visibly beats the v1.12 screen-space composite on the dam-break and
thin-sheet captures; otherwise the plan exits at the experiment and we keep screen-space.

## The reversal this plan undoes — read first

[`decisions/rendering.md`](../decisions/rendering.md) and
[`architecture/rendering.md`](../architecture/rendering.md) §"Removed surface path" state
that an extracted-surface renderer (marching-cubes modules, shaders, host tables,
settings, web controls) was **deliberately removed**: "there is no extracted-surface
compatibility path." This plan brings it back. That's a legitimate change of direction,
but it must be a *decision*, not an accident — the ship-time migration **must** rewrite
those two sections, not just append to them.

## De-risk gate (do this before building the full pipeline)

1. Implement **occupancy-boundary quads only** (the source plan's Mode 1 / emergency
   fallback): generate quads on Liquid/Air cell boundaries, feed them through the v1.12
   material. ~½ day, throwaway-quality.
2. Capture Scene 3 (dam break) and Scene 5 (thin sheet) with **quads vs the v1.12
   refractive composite**.
3. **Decision point:** if the blocky-but-coherent surface already reads better than
   screen-space, invest in dense marching cubes (below). If it doesn't, stop here, delete
   the experiment, and the series skips to v1.15 with screen-space as the surface. Record
   the call in the roadmap.

This answers the open strategic question cheaply instead of spending days building MC on
faith and discovering it doesn't beat what we already shipped in v1.12.

## Scope (if the gate passes)

**In:**
- Dense render-density grid (sized from the *actual* `nx·ny·nz`, not the source plan's
  4M/2M caps — re-derive). Fixed-point particle splat consistent with the P2G
  `scatter.wgsl`/`normalize.wgsl` integer-atomics pattern.
- Density smoothing pass (1–4 iterations, tunable).
- Marching cubes over the dense grid → vertex/index (or indirect) buffers with GPU
  counters; density-gradient normals.
- `HeroSurfaceSource` Live enum: `ScreenSpace | MarchingCubes` (default stays
  `ScreenSpace`; MC is opt-in). The mesh feeds the **same** composite material.
- Optional **mesh front/back depth thickness** as an alternate thickness source (the
  source plan's thickness "track D", folded in here since it only matters with a mesh).
  Existing screen-space particle thickness stays the default.
- Debug views: `DensityGridSlice`, `ActiveSurfaceCells`, `MeshNormals`, `MeshWireframe`,
  `MeshTriangleCount`.

**Out:** SDF-field MC and active-cell compaction (follow-ups, source-plan Mode 3/active);
adaptive/sparse grids (explicitly deferred); anything that requires the mesh to be
watertight before the rest of the series works.

## Watch-outs

- **Keep screen-space as the always-available fallback.** If MC produces garbage on a
  scene, the water view must degrade to the v1.12 path, not to nothing.
- **No readback.** Vertex/index buffers + counters live on the GPU; draw via indirect.
  Triangle-count for the stats overlay comes from the indirect buffer or a *throttled*
  timing read — never a per-frame map.
- **Numbers re-derived from the tank.** 64³ render grid = 262k cells → at most a few
  hundred k triangles; the 4M-vertex caps in the source plan signal it wasn't sized to
  this `[-1,1]³` box.
- **Blobby vs blocky tuning** is the main quality risk: iso value, splat radius, smoothing
  iterations, mesh Laplacian. Expose them all Live; thin sheets are the hard case (Scene
  5) — they should not balloon into giant blobs.
- **Perf.** Splat + smooth + MC + draw on top of the v1.12 passes at ~254k particles in
  WASM/WebGPU is the series' heaviest single addition. Profile it; this is the most likely
  place to blow `target_frame_ms` and need the screen-space fallback under dynamic quality.

## Settings (Water tab → Surface group, all Live)

Mirror `[render.hero_water.surface]`, flattened: `surface.source` (enum),
`render_grid_scale`, `particle_splat_radius_cells`, `particle_splat_strength`,
`density_fixed_point_scale`, `iso_value`, `smoothing_iterations`, `smoothing_strength`,
`normal_mode`, `normal_smoothing`, `mesh_laplacian_iterations/strength`,
`max_vertices`/`max_triangles` (re-derived), `fallback_to_screen_space`,
and the mesh-thickness toggles from `[render.hero_water.thickness]`.

## Exit gate

- De-risk gate recorded (passed → built MC; failed → stopped, roadmap notes the call).
- If built: Scene 3 + Scene 5 captures show the mesh surface beating the v1.12 composite;
  thin sheets don't become blobs; `HeroSurfaceSource` switches live between screen-space
  and mesh.
- WSL build + `cargo test --lib` green (add host-side MC table tests if a host reference
  is reintroduced).
- `gpu.render_ms` with MC on/off recorded; fallback-to-screen-space verified to still work.
