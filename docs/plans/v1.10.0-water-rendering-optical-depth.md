# v1.10.0 - water rendering optical depth

## Mission

Improve how the simulated fluid reads as water by implementing the first direction
selected in [`water-rendering-research.md`](water-rendering-research.md):
**particle-native optical-density billboards** in the existing particle renderer.
Done means the default particle visual has transparent thin edges, denser overlap,
less bead-like shading, heavy live configuration, and matched before/after capture
plus profiler evidence.

## Scope

In scope:

- Baseline captures of the current particle renderer before code changes.
- `ParticleRenderer` / `particles.wgsl` changes that replace flat alpha with
  optical-density/transmittance behavior per particle.
- A settings-registry change that renames the main transparency control away from
  opacity semantics.
- An explicit particle depth-state change or justification. The expected first pass
  disables particle depth writes while preserving depth testing, so transparent overlap
  can contribute; if depth writes remain on, the implementation must prove why.
- Existing-pass implementation only: no offscreen accumulation textures, no composite
  pass, and no new persistent render buffers unless this plan is updated first.
- Configurable water-look controls rather than one hardcoded style.
- Improvements for both low-count density readability and high-count visual quality
  where practical.
- Before/after captures at the same scene, color, particle count, and camera angle.
- Honest profiler notes using the current `gpu.render_ms` timing source.

Out of scope for v1.10:

- Reviving the old marching-cubes path or restoring removed MC modules/shaders/settings.
- A full smooth surface renderer.
- Weighted blended transparency, screen-space density splats, blur/composite paths, or
  voxel volume rendering. These are follow-ups if v1.10 captures show the same-pass
  approach is insufficient.
- Source/drain, particle allocation changes, or simulation physics changes.
- Particle dispatch/scale work.
- New CPU/GPU readback outside the existing throttled profiler/capture path.

## Approach

### Expected code surfaces

Implementation is expected to touch only these product surfaces unless evidence forces
an explicit plan update:

- `crates/fluid-lab/src/settings/mod.rs` - setting rows, accessors, color/slider hints
  if needed.
- `crates/fluid-lab/src/lib.rs` - live `set_setting` dispatch for renamed/new render
  settings.
- `crates/fluid-lab/src/gpu/mod.rs` - stored particle-look fields and recreate
  reapplication.
- `crates/fluid-lab/src/gpu/particles.rs` - particle uniform, blend/depth state, live
  setters.
- `crates/fluid-lab/src/gpu/shaders/particles.wgsl` - optical-density alpha and
  shading behavior.
- `web/panels.js` only if the existing schema-driven panel cannot render the new
  labels/options from `config_json`.

Do not touch simulation physics, particle allocation, dispatch tiling, or old
surface-extraction files for this plan.

### Implementation steps

1. **Capture baseline first.** Run real-GPU captures for the current renderer before
   any code change. Keep the PNG and console artifacts.
2. **Rename opacity semantics.** Add `render.water_optical_density` as a Live Render
   setting and use it as the main transparency control. Remove or retire
   `render.particle_alpha`; if it remains for compatibility, it must not be labeled as
   the default water opacity control when the shader interprets density.
3. **Keep the heavy-config surface.** Preserve live controls for particle size, edge
   softness, sphere shading, slow/fast colors, and speed scale. Initial default target:
   `render.water_optical_density = 1.25`, `render.particle_shading = 0.25`, existing
   particle size/edge/colors unchanged unless the baseline/after captures justify a
   different default.
4. **Change particle shading semantics.** In `particles.wgsl`, derive alpha from a
   sphere-like per-fragment thickness and optical density, e.g. thin rims have low
   alpha and centers have higher alpha. Keep speed color and shading configurable.
5. **Fix transparent overlap behavior deliberately.** Change the particle pipeline so
   transparent particles can contribute through depth. Expected first version:
   `depth_write_enabled = false` for particles, while preserving a depth test against
   already-written scene depth. Document any slice-overlay side effect in the ship
   notes if it is visible.
6. **Do not add a render pass in v1.10.** If implementation needs weighted blended
   transparency or a composite target, stop and update this plan with the new targets,
   resize behavior, timing sections, and exit gates before coding that path.
7. **Capture after with identical cases.** Repeat the baseline matrix and record the
   visual outcome plus profiler deltas.

Implementation should preserve the app's lab identity. A prettier default is allowed,
but inspectable/debug modes and heavy configuration remain product requirements.

## Settings contract

Required controls:

| Setting | Panel | Apply | Contract |
|---|---|---|---|
| `render.water_optical_density` | Default / Render | Live | New main transmittance control. F32 range 0..8, default 1.25 unless capture evidence says otherwise. |
| `render.particle_size` | Default / Render | Live | Existing visual radius control; still render-only and not mass. |
| `render.particle_edge` | Default / Render | Live | Existing soft-edge control; must remain usable for spray/rim tuning. |
| `render.particle_shading` | Default / Render | Live | Existing sphere-shading strength; default should avoid bead-like water while staying configurable. |
| `render.particle_slow_color` / `render.particle_fast_color` | Default / Render | Live | Existing packed RGB colors for water tint and speed highlight. |
| `render.speed_scale` | Default / Render | Live | Existing speed ramp scale, independent from optical density. |

Do not expose a control named "opacity" if its value feeds an optical-density or
Beer-Lambert-style equation. Old persisted `render.particle_alpha` values may be
ignored, migrated, or treated as a legacy debug value, but the UI and docs must not
make the semantic lie.

## Evidence and profiler requirements

Minimum before/after capture matrix:

| Case | Capture intent | Required artifact facts |
|---|---|---|
| Default count, default scene | Normal product look and cost. | PNG + console, requested/actual particles, grid dims, pressure iterations, render settings, `timing`, `gpu.render_ms`, frame p50/p95. |
| Low count, e.g. `PARTICLES=32768` | Shows whether gaps read denser without opaque discs. | Same as default; same particle request before/after. |
| High count, e.g. `PARTICLES=2000000` if accepted | Shows whether higher-count water looks smoother and whether render cost stays honest. | Same as default; if rejected or too slow, record the `stats_json` rejection/skip instead of claiming a result. |

Capture rules:

- Use the static path served by `app/run.sh` at `http://localhost:5184/`.
- Use `app/tools/capture.mjs` through Windows Chrome as documented in
  `agent-context/build-run.md`; WSL node is not valid for the real-GPU capture.
- Use the same scene, camera, colors, particle count, wait time, pressure setting,
  interaction state, and panel state for each before/after pair.
- Keep the final machine-readable `stats_json` line from each console artifact.
- A screenshot of the unsupported overlay is a failed capture, not evidence.

Profiler rules:

- Prefer `gpu-timestamp` samples. If unavailable, label the fallback and do not claim a
  precise GPU render-cost delta.
- For the recommended same-pass path, report `gpu.render_ms` before/after and frame
  p50/p95. Treat a render regression greater than either about 0.5 ms or 20% at the
  default case as a finding that needs explanation or user acceptance.
- If a new pass/texture/buffer becomes necessary, update this plan before coding it and
  add a timing/reporting requirement that separates particle accumulation from
  composite cost.

## Exit gate

- `water-rendering-research.md` has a visual target, candidate comparison, and chosen
  first implementation direction.
- Baseline captures exist before the implementation change.
- The implementation exposes the agreed water-look controls through the settings
  registry and live bridge.
- `render.particle_alpha` no longer appears as a misleading default opacity control if
  the shader uses optical-density semantics.
- The particle renderer remains GPU-native and does not add normal-frame readback.
- The implementation stays in the existing render pass, or this plan is updated before
  any multi-pass/offscreen work starts.
- Any new pass/texture/buffer/timing cost is measured and documented if the scope is
  deliberately expanded.
- `cd /home/adamg/fluid-simulation/app && cargo build --target wasm32-unknown-unknown`
  passes.
- `cd /home/adamg/fluid-simulation/app && cargo test --lib` passes, especially if
  settings registry behavior changes.
- Real-GPU captures compare before/after at the same scene and camera conditions.
- The profiler note is honest about render cost and timing source.

## Discipline rules

- Do not restore old marching-cubes code by default.
- Do not make opacity controls lie; rename semantics if needed.
- Do not hide solver problems behind an unconfigurable cinematic material.
- Do not update architecture or decisions docs from this planning change alone.
- Do not update architecture or decisions docs until implementation is underway or
  shipped, and then update only the docs named by the touched surfaces.
