# v1.3.0 - scale performance and profiler

## Mission

Make scale the headline. After marching cubes is removed, measure why high particle
counts slow down or appear frozen, improve profiler visibility where needed, and ship
the highest-confidence work that lets the app show more convincing water volume on the
user's high-end PC. Done means performance claims are hardware-dependent and backed by
raw measurements, with 30 FPS considered a good target for heavy runs.

## Scope

In scope:

- Build a real-GPU measurement matrix on the user's PC after MC removal.
- Test particle counts around the current interesting bands: 2M, 4M, 8M, and any
  extra points needed to find the cliff.
- Record grid dimensions, particle count, pressure iterations, render mode, timing
  source, frame time, dropped sim time, buffer memory, browser/device-limit messages,
  and top profiler sections.
- Improve profiler UI/logging if it does not make those facts obvious.
- Investigate both routes to "more water": more real particles and denser-looking
  rendering from fewer particles.
- Ship the top measured wins when they are clear enough and scoped enough for this
  phase. Candidate areas include particle rendering cost, render decimation/LOD,
  point-rendering alternatives, particle transfer bottlenecks, allocation behavior,
  timing/drop policy visibility, and tier presets.

Out of scope:

- Universal FPS promises. The claim is hardware-dependent until broader hardware is
  tested.
- Optimizing CG or transfer code without profiler evidence.
- Source/drain, wall-contact fixes, or broad UI copy polish.
- MC-on measurements; MC should be gone.

## Approach

This phase has an audit wave followed by one or more implementation waves. Read-only
measurement prep can run separately, but code edits stay sequential per repo rules.

### Wave A - measurement and profiler audit

- Add or improve the cheapest tooling needed to capture repeatable scale facts:
  console stats, `stats_json`, profiler panel rows, capture harness `EVAL` snippets, or
  a small documented manual matrix.
- Confirm the profiler states timing honestly (`gpu-timestamp`, fallback, or
  unavailable) and exposes enough structural facts for scale work.
- Run a matrix with MC absent:

| Particles | Grid | FPS target | Pressure iters | Detailed profiling | Expected purpose |
|---|---|---|---|---|---|
| 2M | current default and any chosen tier | 30 | default | off/on as needed | baseline high-water run |
| 4M | same | 30 | default | off/on as needed | practical upper target |
| 8M | same | 30 | default | off/on as needed | cliff/stress target |

Add rows if the cliff lands between these values. Keep raw output; do not summarize it
away.

### Wave B - choose the bottleneck class

Classify the limiting factor from evidence:

- **Particle render bound** - many billboard vertices/fragments dominate frame time.
- **Particle transfer bound** - mark/P2G/G2P/advect/impulse scale linearly or suffer
  atomic contention.
- **Grid/pressure bound** - classify/divergence/CG/gradient dominates at chosen grid.
- **Allocation/device-limit bound** - buffers, max sizes, memory budget, or browser
  watchdog explain the cliff.
- **Timing policy bound** - the app drops sim time or renders stale frames, so it looks
  frozen while the loop survives.

### Wave C - implement the clearest wins

Pick the smallest set that the measurements justify. Examples:

- Better profiler presentation for heavy runs: visible timing source, top costs,
  dropped-time counter, memory, and scale facts in one compact section.
- Particle render alternatives or decimation if rendering dominates. Accept fewer real
  particles if the visual density is more convincing.
- Tier presets only after measurements define useful tiers. Keep targets as
  hardware-dependent.
- Particle transfer or allocation work only when profiler/device evidence points there.

If the winning optimization is too large for this phase, write a follow-up plan and
ship the measurement/profiler improvements first.

## Exit gate

- A raw measurement table exists for at least 2M, 4M, and 8M particles on the user's
  PC, with the fields named above.
- The report identifies the 8M behavior as allocation, render, compute, timing policy,
  adapter/browser limit, or a mix.
- At least one profiler/scale improvement ships, or the plan records why the measured
  top fix is too large and creates the next implementation plan.
- `cd /home/adamg/fluid-simulation/app && cargo build --target wasm32-unknown-unknown`
  passes for code changes.
- `cd /home/adamg/fluid-simulation/app && cargo test --lib` passes for code changes.
- Any visible profiler/rendering changes have a real-GPU capture.
- `decisions/performance.md` reflects measured, hardware-dependent claims.

## Discipline rules

- No performance claim without raw profiler or capture output.
- Keep 30 FPS as the heavy-run target unless measurements show it is unrealistic for a
  chosen tier.
- Do not reintroduce MC, a surface replacement, or a hidden expensive render path.
- Prefer changes that make fewer particles read as more water and changes that allow
  more particles; both serve the same product goal.
