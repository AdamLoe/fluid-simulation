# v1.x - particle-scale orchestrator

## Mission

Turn the June 7 planning/audit docs into one executable map. The product direction is
now particle/liquid-cell-first: remove marching cubes completely, make scale and
profiling the first priority, then improve simulation feel, then simplify the settings
surface without hiding expert power. Done means every subplan below has shipped, the
architecture/decisions docs have been rewritten to the resulting current state, and the
old draft plans can be deleted.

This is a planning/orchestration doc, not a code implementation. The orchestrator who
picks this up should read this file first, then only the subplan for the active phase.

## Locked decisions

| ID | Decision | Consequence |
|---|---|---|
| D1 | Performance first, simulation feel second, polish third. | Do not spend the first implementation wave on UI copy or interaction toys. |
| D2 | Particles and liquid-cell/voxel views are the app's first-class visual identity. | Marching cubes is removed, not hidden, polished, or kept as a dev feature. |
| D3 | The Mesh toolbar button goes away. | Remove the UI control, URL toggle, WASM bridge, settings, runtime path, shaders, and docs. |
| D4 | Scale target is hardware-dependent and measured on the user's high-end PC; 30 FPS is acceptable. | Claims must cite raw profiler/capture evidence, not a fixed universal particle-count promise. |
| D5 | The goal is more convincing water volume by both routes: more particles when possible and better visual density from fewer particles. | Performance work may optimize particle count, rendering density, or both. |
| D6 | Wall behavior should be free-sliding with zero stickiness, no bounce/restitution, and no special floor/ceiling model. | Fix wall/ceiling cling without adding negative pressure, bounce, or orientation-specific wall semantics. |
| D7 | Source/drain should eventually create/destroy particles, but is future roadmap work. | Current interaction work is random tank rotation plus impulse wave making. |
| D8 | Settings may have 0, 1, or 2 help fields; functional and technical help appear adjacent with distinct styling. | Add optional schema fields rather than parsing one long tooltip string. |
| D9 | Advanced controls can be collapsed, but nothing should become a hidden internal. | Keep expert/debug access available; default organization can be opinionated and revisited. |
| D10 | No kill switches as product strategy. | Implement the chosen direction; fallbacks should stop and report a blocker, not leave both product paths exposed. |

## Dependency map

- **Remove MC before scale work.** MC settings, lazy allocation, shaders, and the Mesh
  button would pollute performance measurements and settings copy.
- **Measure before promising scale.** v1.3.0 should produce raw profiler/capture facts
  before choosing optimization work or public claims.
- **Fix simulation semantics before final settings copy.** v1.4.0 defines wall contact,
  volume correction, and compactness behavior; v1.5.0 then explains and groups it.
- **Run interaction work after wall/volume quality.** Random tank rolling and impulse
  wave making make wall cling easier to notice, so they should not ship first.
- **Source/drain stays future.** It creates/destroys particles and needs its own mass
  mutation/boundary model later; it should not block v1.6.0.

## Orchestrator operating rules

Use [`../agent-context/orchestrating.md`](../agent-context/orchestrating.md) and the
generic orchestrating rules. This repo has no git/worktree isolation for sub-agents, so
code-touching agents run sequentially. Read-only audits can run in parallel.

The orchestrator holds the map, not the implementation details:

- Keep this streams table current from observed agent results, not optimism.
- Do not read broad source files in the hub context unless deciding a routing issue.
- For each phase, dispatch against the matching subplan and the smallest architecture
  docs named there.
- Require each implementation agent to report files changed, exact gate command and
  result, deferrals, and doc updates needed.
- Run one consolidated final browser capture for the whole visible product, plus
  targeted captures for visible wins where the subplan asks for them.
