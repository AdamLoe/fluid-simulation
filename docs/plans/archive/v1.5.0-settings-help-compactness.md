# v1.5.0 - settings help and compactness model

## Mission

Make the settings panel easier to scan without removing expert power. Settings may
have no help, functional help only, or functional plus technical help. Technical help
appears adjacent to functional help with distinct styling, not buried in one long
hover paragraph. Done means the registry schema, panel rendering, grouping, and copy
match the final post-MC, post-wall-fix simulation model.

## Scope

In scope:

- Add optional two-layer help to the settings registry, such as `tooltip` for
  functional help and `technical_tooltip` for technical help. Either field may be
  absent.
- Render functional and technical help side by side or immediately adjacent, with
  distinct color/treatment.
- Delete obvious help rather than rewriting every setting. Examples: particle opacity,
  simple color components, and axis/component controls whose label is enough in
  context.
- Group settings into a compact default view plus collapsed Advanced/Dev sections.
  Nothing becomes truly hidden; expert/debug controls remain reachable.
- Keep `solver.pressure_iterations` visible because it is important to FPS and physics.
- Move `classify.liquid_threshold` and `classify.surface_dilation` to Advanced unless
  v1.4.0 proves they deserve default placement.
- Preserve most knobs initially. The question is what they do and how they should be
  explained, not a broad removal pass.
- Remove all MC help/settings assumptions. MC should already be gone in v1.2.0.

Out of scope:

- Changing simulation behavior.
- Reintroducing MC settings or old surface controls.
- Implementing source/drain or wave tools; v1.6.0 owns interaction controls.
- Finalizing a forever-perfect basic/advanced split. The user expects to revisit it
  after living with the app.

## Approach

### Wave A - schema and serialization

- Extend `Setting` with optional technical help while allowing 0, 1, or 2 help fields.
- Update `config_json` so absent help fields stay absent rather than emitting empty
  strings.
- Keep apply class, validation, defaults, category, and display metadata stable.
- If backward compatibility matters for panel code, keep `tooltip` as the functional
  field and add `technical_tooltip`.

### Wave B - panel rendering

- Render help affordances next to each other. Functional help should read as the quick
  "what changes?" answer; technical help should read as "how does the sim use this?"
- Use different color/treatment for the two help types so users can skim. The exact
  visual style belongs to implementation, but it should be obvious which is basic and
  which is technical.
- Support rows with no help cleanly. Do not show empty icons.
- Collapse Advanced/Dev groups by default while keeping them discoverable.
- Keep mobile/narrow behavior usable: adjacent can wrap, but the row must not become a
  wall of prose.

### Wave C - setting taxonomy

Use this starting split, then adjust from v1.4.0 facts:

| Group | Default? | Notes |
|---|---|---|
| Scene and reset-scale controls | Default | Scenario, grid dimensions, particle count. Heavy Reset-class controls need concise warnings. |
| Visual particle controls | Default | Particle size, speed scale/color, opacity/shading if still product-facing. Obvious controls can have no help. |
| Core physics feel | Default | Gravity, FLIP blend, wall friction, CFL if the user tunes feel often. |
| Solver quality | Default | Pressure iterations stays visible because it trades FPS and water solidity. |
| Compactness/volume internals | Advanced | Rest density, volume stiffness, drift clamp unless v1.4.0 makes one a primary feel control. |
| Classification controls | Advanced | Liquid threshold and surface dilation are powerful but technical. |
| Camera/presentation | Default or collapsed group | Keep controls if they affect presentation; remove mesh-only sun controls if v1.2.0 did not already. |
| Dev/profiler | Collapsed Dev | Detailed profiling and any remaining debug-only controls. |

### Wave D - copy policy

- Functional help: one short sentence about visible effect.
- Technical help: one compact technical paragraph or short sentence naming math,
  apply class, and failure mode where useful.
- No help for self-evident controls.
- Do not paste architecture docs into UI copy. Use the same terms and link/update docs
  for durable explanations.

## Exit gate

- `config_json` supports settings with 0, 1, or 2 help fields.
- The settings panel renders functional and technical help adjacent with distinct
  styling and no empty affordances.
- Advanced/Dev grouping is collapsed by default and discoverable.
- MC settings/help are absent.
- Pressure iterations remains visible.
- At least one before/after capture or browser screenshot shows the panel is lighter.
- `cd /home/adamg/fluid-simulation/app && cargo build --target wasm32-unknown-unknown`
  passes.
- `cd /home/adamg/fluid-simulation/app && cargo test --lib` passes if registry tests or
  serialization-sensitive code changed.

## Discipline rules

- Do not hide internals permanently; collapse and explain them.
- Do not remove physics knobs just because they are confusing. First make their role
  clear.
- Do not produce another one-string tooltip convention.
- Do not write version-flavored notes into architecture docs; rewrite current state.
