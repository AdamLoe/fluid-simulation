# v1.9.0 - particle performance follow-up

## Mission

Optimize the measured bottleneck after tiled particle dispatch has produced a fresh
scale matrix. This plan is intentionally a placeholder until
[`v1.8.0-particle-dispatch-tiling.md`](v1.8.0-particle-dispatch-tiling.md) records the
new bottleneck and updates this doc with evidence.

## Scope

Potentially in scope after v1.8.0 evidence:

- Transfer/scatter/G2P optimization.
- Render cost optimization.
- Pressure/grid optimization.
- Memory/allocation pressure fixes.
- Timing/profiler policy improvements if measurement itself is the blocker.

Out of scope unless the user promotes it:

- Render decimation/LOD for high counts.
- Particle-count presets.
- Water-look rendering.
- Source/drain or new physics features.

## Approach

Do not start implementation from this placeholder. First, v1.8.0 must update this doc
with:

- the exact scale rows measured;
- the sorted GPU costs;
- the selected optimization class;
- the candidate change;
- before/after evidence required.

Once updated, keep the implementation narrow to the selected class. If the top
bottleneck is ambiguous, write a small measurement spike instead of rewriting multiple
subsystems.

The legal particle ceiling work succeeded: requested 8M submits as tiled
`65,535 x 2 x 1` instead of failing on the old one-dimensional dispatch ceiling.

Selected optimization class: **transfer/scatter/G2P measurement spike and narrow
optimization**.

Reasoning:

- At 8M, render is the single largest measured bucket, but the current plan keeps
  render decimation/LOD out of scope unless the user promotes it. v1.10 is already the
  active renderer-quality stream, so v1.9 should not bundle a renderer policy change
  into this pass.
- Scatter cost scales directly with particle count and is the largest in-scope,
  particle-scale cost across 4M/8M (`13.694 ms` and `22.604 ms` summed over U/V/W).
- G2P is also particle-linear and measurable (`6.961 ms` at 4M, `8.575 ms` at 8M),
  but it carries wall-aware sampling invariants, so any implementation must be narrow
  and backed by before/after captures.

Candidate change:

- First, inspect scatter/G2P shader/runtime paths for a low-risk reduction in redundant
  work or memory traffic without changing fixed-point accumulation, wall-aware G2P,
  or particle behavior.
- If no low-risk patch exists, ship v1.9 as a measurement conclusion: transfer paths
  are the selected bottleneck and deeper changes need a dedicated plan.

Before/after evidence required:

- Repeat the 4M and 8M captures with `DETAILED=1` and the same scene/camera/settings.
- Report scatter U/V/W, G2P, render, sim, p50/p95/p99, dropped time, and dispatch
  shape before/after.
- Treat any visual or wall-contact regression as a failed optimization.

## Focused transfer-path review - 2026-06-08

Current code review found one safe, narrow transfer-path optimization:

- `scatter.wgsl` and `g2p.wgsl` converted particle world-to-grid coordinates with
  scalar divides by `h` even though `Params.geom.y` already carries `inv_h`.
- The host initializes `geom` as `[h, 1.0 / h, dt, FIXED_SCALE]`; with the current
  uniform cell size `h = 2/64`, `inv_h = 32.0` is exactly representable.
- Replacing `/ h` with `* inv_h` removes per-particle divide work in the particle-linear
  transfer passes without changing fixed-point scatter atomics, bounds guards,
  wall-aware G2P static-solid face skipping, PIC/FLIP blend, CFL clamp, wall friction,
  RK1 advect, deterministic recovery, or dispatch shape.

Implemented patch:

- `app/crates/fluid-lab/src/gpu/shaders/scatter.wgsl`: use `params.geom.y` for the
  P2G grid-coordinate conversion.
- `app/crates/fluid-lab/src/gpu/shaders/g2p.wgsl`: use `params.geom.y` in all three
  MAC face samplers.

Outcome:

- The shader-only `inv_h` change produced a measurable transfer-path win at both 4M
  and 8M without changing the dispatch shape or preflight result.
- Remaining high-scale cost is now shared between render and transfer. Render remains
  out of scope for v1.9 and belongs to either the water-rendering stream or a future
  render-decimation/LOD plan if the user promotes that work.
