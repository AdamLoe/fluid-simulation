# v1.11.0 - screen-space lit water thickness

## Mission

Make deep water read as a lit, voluminous liquid body while thin water (spray, wall
films, shallow puddles) stays see-through. v1.10 put optical density on the
per-particle billboard; the shipped result correctly clears thin water but flattens
large volumes into a uniform tinted blob. The cause is structural: a single billboard
cannot know how much water sits behind it, and unsorted per-particle shading averages
every highlight into mush. This plan moves opacity and lighting to **screen space**:
accumulate per-pixel water thickness, reconstruct a smooth front surface, and composite
a Beer-Lambert tint plus lighting over the existing scene.

The visual goal is **staged so each done-bar is honest** (see Staging):

- Stage 1 reads as a depth-correct tinted fill — infrastructure + correctness, **not**
  water yet.
- Stage 2 reads as a **lit liquid body** — smooth shaded surface, but still gel-like.
- Stage 3 is a follow-up refraction/backdrop pass. Refraction is the cue that separates
  water from lit jelly, but it needs visible scene detail to distort and a sampleable
  offscreen scene-color target. v1.11 does **not** claim "reads as water" unless that
  follow-up scope is explicitly pulled back in.

This plan deliberately reopens the "rendering stays single-pass" decision in
[`decisions/rendering.md`](../decisions/rendering.md). That decision's own *revisit-when*
clause — "a measured, approved rendering feature requires an offscreen or multi-pass
composition path" — is the trigger; v1.10's captures and the flat-volume report are the
measurement.

## Pre-implementation contracts

This is the compact contract an orchestrator should resolve before assigning code work.
The derivation and rationale are below, but these facts should not vary by subtask.

- **Scope.** v1.11 ships Stage 1 + Stage 2: count-independent thickness, smoothed
  front-depth normals, and lit screen-space liquid. Stage 3 refraction is a follow-up
  unless the same implementation pass also adds visible opaque scene detail to refract.
- **Thickness units.** The thickness target stores approximate world-space path length.
  Each splat writes `kernel_len * volume_scale`, where
  `kernel_len = 2 * radius_world * sqrt(max(0, 1 - dot(uv, uv))) * edge(uv)` and
  `volume_scale = particle_volume / kernel_volume`. `kernel_volume` is the integral of
  the same kernel over the billboard disc; for the hard-sphere kernel it is
  `(4/3) * pi * radius_world^3`. If the edge taper changes that integral materially,
  compute the matching normalization on the CPU and upload it with the particle camera
  uniform. `particle_volume` is derived from the represented liquid volume divided by
  actual particle count, not tuned per capture.
- **Particle size semantics.** `render.particle_size` may change raster coverage and
  smoothing feel, but the normalization above must preserve represented volume. It must
  not be a hidden opacity control.
- **Nearest depth.** The nearest-Z MRT target stores the front sphere surface in
  positive eye-space distance, not the particle center or billboard plane. The shader
  needs the view transform (or an equivalent camera-forward/center-eye contract), not
  only `view_proj`: `nearest_z = -(center_eye.z + radius_world * nz)` under the current
  right-handed eye-space convention. If that convention changes, write it down next to
  the shader.
- **Target formats.** Use `R16Float` for Thickness and Nearest-Z; do not use `R32Float`
  blending or sampled `Depth32Float` unless the fallback path is explicitly activated.
- **Timing.** Keep one public `gpu.render_ms`, but make it span every render pass. The
  current `render_writes()` attaches one begin/end pair to one render pass; the new path
  must split that helper into begin-only on the first render pass and end-only on the
  final render pass, or use an equivalent same-shape timing contract.
- **Stage 3 prerequisite.** Refraction requires both a sampleable offscreen scene-color
  target and visible opaque scene detail (for example a subtle floor/back-wall grid).
  Do not allocate the scene-color target in v1.11 unless refraction is in scope.

## Why the current approach can't be tuned into the goal

- **No accumulated depth.** `a = 1 - exp(-density * thickness)` in
  [`particles.wgsl`](../../app/crates/fluid-lab/src/gpu/shaders/particles.wgsl) uses one
  billboard's sphere chord. A lone wall particle and the front particle of a deep column
  both have center thickness ≈ 1, so both get the same alpha. Thin water only reads as
  transparent today because *few* particles land there.
- **Shading averages to flat.** With depth-write off and alpha blending, hundreds of
  overlapping transparent billboards blend in arbitrary order; each billboard's diffuse +
  specular cancels into a uniform mid-tone. No coherent surface to light → flat.

Per-pixel accumulated thickness is the missing signal, and it is order-independent
(additive, no sorting). `transmittance = exp(-k · thicknessSum)` is near-1 (clear) for
thin water and near-0 (opaque) for deep water.

## Critical correctness constraints (resolve before coding)

These four are not tuning details; getting them wrong fails pipeline creation or fails
the capture matrix.

### C1 — Thickness is ~linear in particle count; the normalization is load-bearing

A pixel accumulates **every** billboard whose center lies in a fixed-radius cylinder
around its ray (billboard radius is fixed at `crate::sim::H * 0.35`,
[gpu/mod.rs:167](../../app/crates/fluid-lab/src/gpu/mod.rs#L167); it is **not** tied to
particle spacing). The particle count in that fixed-radius cylinder scales with number
density ∝ **N**. So for the *same* water body, the raw chord-sum
`Σ (2·radius·nz·edge)` scales ~linearly with particle count: roughly **~60×** between
the 32k and 2M capture cases. A single un-normalized `k` therefore makes low-count water
nearly invisible and high-count water instantly saturated — the count-dependence
masquerades as a depth result.

`render.particle_size` also leaks into opacity, because `radius` sits inside the
thickness term and `set_radius_scale` ([particles.rs:149](../../app/crates/fluid-lab/src/gpu/particles.rs#L149))
multiplies it.

**Fix (required, not optional):** normalize each billboard kernel to the volume its
particle represents. The fragment writes `kernel_len * volume_scale`, with
`volume_scale = particle_volume / kernel_volume`; `particle_volume` is the represented
liquid volume divided by actual particle count, and `kernel_volume` is the integral of
the same chord/edge kernel over the billboard disc. That makes N particles in a column
sum to approximately the column's true depth regardless of N, and it prevents radius
from becoming a hidden opacity control. The self-consistency check: this 1/N-style
weight only yields an N-independent sum because the raw signal is linear in N; if it
were N^(1/3) the same weight would over-correct to N^(−2/3). The capture matrix (below)
must **actively verify one `k` holds across 32k / default / accepted-high case**, not
assume "close enough."

### C2 — Thickness and nearest-depth need opposite depth states; pick the MRT path

Thickness accumulation needs depth-**write off** so every layer along the ray
contributes. Nearest-surface depth (for Stage 2 normals) needs the closest fragment to
win. You cannot get both from one attachment with depth-write on (a far particle behind a
near one would be rejected and never accumulate thickness).

Also: **`R32Float` is not blendable** in core WebGPU without the `float32-blendable`
feature, and the device requests only `TIMESTAMP_QUERY`
([gpu/mod.rs:95](../../app/crates/fluid-lab/src/gpu/mod.rs#L95)). A `Min`-blend pipeline on
`R32Float` fails creation.

**Resolution (default):** one **MRT thickness pass**, depth-test `Less` against the
shared scene depth, depth-**write off** on both attachments. The pass writes world-space
path length to Thickness and the front sphere-surface positive eye distance to
Nearest-Z:

- `target0 = R16Float`, additive (`src=One, dst=One`) — thickness.
- `target1 = R16Float`, `Min` blend — nearest eye-space distance (Stage 2). Use the
  sphere front depth (`center_eye.z + radius_world * nz`, converted to positive
  distance), not center depth. `R16Float` *is* blendable; `Min` gives nearest-wins
  without depth write.

Fallback only if f16 Z banding survives the smoothing pass: a separate `Depth32Float`
fluid-depth pass (test `Less`, write on → nearest for free), sampled via `textureLoad`.
That is a genuine extra pass; do not budget it unless banding forces it.

`R32Float`/depth targets are **not filterable** without `float32-filterable` — reconstruct
normals with `textureLoad` point fetches, never `textureSample`.

### C3 — Defer scene-color sampling unless refraction is in scope

Stage 1 and Stage 2 do not need to sample scene color. They can draw the opaque scene
directly to the swapchain/depth, accumulate Thickness/Nearest-Z offscreen, then draw a
full-screen composite pass that loads the swapchain and alpha-blends the water result
over it. This avoids allocating a swapchain-sized scene-color target before it has a
visual purpose.

Refraction (Stage 3) is different: it must sample the opaque scene from a **sampleable
offscreen color target** (sample-and-distort, not blend-over-swapchain). That is a real
`render()` spine change and should only land with the refraction follow-up, plus visible
opaque scene detail for the distortion to act on. If Stage 3 is pulled into v1.11, add
Scene color to the target contract and render opaque into that texture before the final
composite.

### C4 — Smoothing is part of the Stage 2 technique

Nearest-Z is built from sphere billboard caps. Without a smoothing pass, depth normals
will light individual splats rather than a coherent front surface. The Stage 2 pass is
therefore not "add normals, then tune blur if needed"; it is **MRT nearest depth +
separable narrow-range/bilateral smoothing + normal reconstruction** as one feature.
Shipping Stage 2 without smoothing means shipping the known speckled failure mode.

The single load-bearing interface for v1.11. Fill/confirm this before any
implementation; C1–C4 are all decided here.

Swapchain is the final output: opaque scene first, water composite over the loaded
swapchain, then the slice overlay. All v1.11 offscreen targets are created at boot and
recreated on `resize` and on the surface `Outdated/Lost` reconfigure branch
([gpu/mod.rs:413](../../app/crates/fluid-lab/src/gpu/mod.rs#L413),
[:717](../../app/crates/fluid-lab/src/gpu/mod.rs#L717)), following the `create_depth`
pattern. None are mapped — no normal-frame readback. The follow-up Scene color target is
listed here only to make the Stage 3 boundary explicit; do not allocate it unless
refraction ships.

## Render-pass structure (the spine)

[`GpuContext::render`](../../app/crates/fluid-lab/src/gpu/mod.rs#L708) becomes:

1. **Opaque pass** → swapchain + shared depth. Clear, draw wireframe (writes depth).
2. **Thickness pass** (MRT) → Thickness + Nearest-Z. See C2. Each particle writes
   `thickness` (normalized per C1) to `@location(0)` and front-surface eye distance to
   `@location(1)`.
3. **Smoothing pass(es)** (Stage 2) → Smoothed-Z. Separable narrow-range / bilateral
   filter; see C4 — mandatory, not conditional.
4. **Composite pass** → swapchain. Loads existing swapchain contents; samples Thickness
   (+ Smoothed-Z in Stage 2). Beer-Lambert tint (Stage 1) → lit surface (Stage 2).
5. **Slice overlay** → swapchain, unchanged.

Stage 3 changes step 1 to render opaque into Scene color, then the final composite
samples Scene color with a normal-driven UV offset before writing the swapchain.

## Decisions baked into this plan

- **D1 — normalize thickness by represented volume (C1).** Required for one `k` to hold
  across the matrix and to stop `particle_size` leaking into opacity.
- **D2 — drop per-particle speed color from the water surface.** Thickness target is
  `R16Float` (thickness only), composite uses a single depth-driven tint (no
  `rgb/max(t,eps)` divide). Speed color survives in the **gated shaded-particle debug
  view** (C-obs), which also satisfies the observability mandate that the slice overlay
  does **not** (the slice shows grid state, not per-particle motion).
- **D3 — depth-based normals are the Stage 2 done-bar; thickness-gradient normals are a
  prototype/fallback.** Thickness-gradient lights integrated depth, not front-surface
  shape, so it cannot deliver "lit voluminous body." MRT nearest-Z (C2) gives depth in
  the same pass at no extra pass cost, so depth normals are also the cheap path. Gradient
  normals are kept only as a quick first-light prototype and as the fallback if depth
  normals stay noisy after smoothing.
- **C-obs — keep the shaded-particle path selectable.** Screen-space water hides
  per-particle motion; the slice overlay does not cover motion. Gate the existing shaded
  fragment path behind `render.particle_view` rather than deleting it, per
  [`decisions/rendering.md`](../decisions/rendering.md) "Visual styling must preserve
  observability."
- **D4 — defer refraction until there is scene detail to refract.** Refraction needs a
  sampleable scene-color target and visible opaque detail. Without that, it risks adding
  pass complexity for a barely visible effect. v1.11 stops at lit liquid unless Stage 3
  is explicitly pulled back in with a backdrop/floor-grid contract.

## Staging

Two increments. Staging is for **risk isolation**: target plumbing and count-correct
opacity first, surface lighting second. **"Done" is the lit result**, not Stage 1.

### Stage 1 — normalized thickness + Beer-Lambert tint

- Build the v1.11 render spine (opaque→swapchain/depth, thickness→offscreen,
  composite→loaded swapchain) **now**.
- Thickness pass writes the **normalized** thickness (D1/C1) to `R16Float`.
- Composite: `T = exp(-k · thickness)`, output water tint with alpha `1 - T` over the
  loaded swapchain.
- Files: `gpu/mod.rs` (targets + 3-pass spine), `gpu/particles.rs` + `particles.wgsl`
  (thickness draw variant + normalized contribution; keep the shaded `fs` reachable via
  the view toggle), new `gpu/composite.rs` + `shaders/composite.wgsl`,
  `settings/mod.rs` + `lib.rs` (`k` semantics, `particle_view` toggle).
- **Done-bar:** deep water is opaque/tinted with a clean depth-correct silhouette; thin
  water see-through; **one `k` verified across 32k / default / accepted-high case**.
  Reads as a flat tint — explicitly **not** "water" yet.

### Stage 2 — lit liquid body (depth normals + mandatory smoothing)

- Thickness pass becomes MRT (add Nearest-Z `@location(1)`, `Min` blend; C2).
- **C4 — smoothing pass is mandatory.** Separable narrow-range/bilateral filter on
  Nearest-Z → Smoothed-Z. In screen-space fluids the blur *is* the technique that turns
  sphere-caps into a surface; without it Stage 2 ships speckled and the blur gets added
  as unplanned scope.
- Composite reconstructs view-space position from Smoothed-Z (`textureLoad`, C2),
  derives the normal, adds diffuse + specular + Schlick fresnel (constant sky tint),
  scaled by `clamp(thickness · shadeGain)` so thin water stays unlit/clear.
  `render.particle_shading` drives strength.
- **Done-bar:** deep water reads as a smooth lit curved surface (highlight + fresnel
  rim), thin water unchanged from Stage 1. This is the **v1.11 shippable visual**: much
  better than v1.10, but still closer to lit lab liquid than cinematic water.

### Follow-up Stage 3 — reads as water (cheap refraction)

- Add visible opaque scene detail first (subtle floor/back-wall/grid) so distortion is
  inspectable rather than theoretical.
- Render the opaque scene into a sampleable Scene color target, then composite to the
  swapchain with a UV offset ∝ surface-normal.xy · thickness, strength
  `render.water_refraction`. **Clamp/guard the edge halo** (sampling background outside
  the silhouette is the standard SSF artifact).
- Constant sky/fresnel tint only — no environment reflections, no caustics (out of scope).
- **Done-bar:** background visibly distorts through the water; the result reads as water,
  not jelly. Only this follow-up may claim "reads as water."

## Out of scope

- Marching cubes / any extracted-surface mesh path.
- Particle physics, allocation, dispatch/tiling, simulation changes.
- Environment reflections, caustics, real (non-fake) refraction depth.
- Foam/spray classification beyond what thickness gives for free.
- New per-frame readback. Offscreen targets are GPU-only.

## Orchestration

**Sequential.** The dependency chain is linear and meets in `GpuContext::render` (the
spine): target plumbing → thickness draw variant → composite consumer → MRT + smoothing.
Do **not** split a "composite stream" from a "plumbing stream"; both edit `render()` and
would conflict in the one load-bearing function. The texture contract above is the
interface, but it grows between stages (Nearest-Z, Smoothed-Z), so it is not a stable
parallelization boundary either. Stage 3 refraction is a later sequential pass because
it changes the opaque-scene destination and the composite input contract again.

## Profiler

- Keep **one coarse `gpu.render_ms`** spanning opaque → thickness → smoothing →
  composite → slice. The public `Readout` shape stays unchanged, but
  `render_writes()` ([timing.rs:282](../../app/crates/fluid-lab/src/gpu/timing.rs#L282))
  cannot stay as one begin/end pair attached to one render pass. Split it into
  begin-only writes for the first render pass and end-only writes for the final render
  pass (or an equivalent helper pair using the same two query slots). Do **not** report
  a timing number that only covers one of the new passes.
- Do **not** claim a per-pass timing split. If we later decide per-pass timing is worth
  it, add `gpu/timing.rs` as an owned surface (→ `profiler.md` per the manifest
  change→doc table) in a follow-up.
- Report `gpu.render_ms` and frame p50/p95 before/after. The thickness pass is ~a second
  particle draw plus full-screen passes — treat a regression beyond ~1 ms or 25% at the
  default case as a finding needing explanation or acceptance.
- Record offscreen-target memory in the render memory note: Thickness + Nearest-Z +
  Smoothed-Z (`w·h·2 B` each). If Stage 3 is pulled in, add Scene color (`w·h·4 B`) to
  the same note. This breaks the v1.2/v1.10 invariant that no offscreen water targets
  exist — update `gpu-resources.md` at ship.
