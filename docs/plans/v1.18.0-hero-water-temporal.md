# v1.18.0 — Hero water: temporal stabilization

> **SHIPPED (2026-06-09, branch `hero-water-14-18`).** History-blend + hard camera-reset
> (NOT reprojection — there is no motion-vector infra; `jitter_enabled` stays off). New
> `TemporalSystem` (gpu/temporal.rs) two-texture ping-pongs the real screen-space targets:
> `thickness_view`, `smooth_z_view` (front-depth — feeds the derived surface normal; there is
> no stored normal target since v1.14 MC was abandoned), and `whitewater_view` (= the
> screen-space "foam" signal; the v1.13 diffuse particles are geometry, not a blendable
> target). The v1.16 caustics stopgap blend is **unified** under `render.hero.temporal.*` (its
> two old caustics-temporal settings removed). Camera-delta uses the model-free `eye_to_world`
> (not the model-baked `view_proj`); hard reset on rotation **and** translation over threshold;
> depth/normal validity rejects. Blend inserted after smooth-y and before caustics-gen so
> stabilized targets feed caustics + composite. Opus review caught **3 must-fix** bugs (ping-
> pong/fixed-binding desync → per-frame rebind; reject wired to raw not history; reset missed
> translation) + a runtime WGSL uniform-control-flow error — all fixed. Valid static + orbit
> on/off A/B (EVAL echo present); **no gross artifacts, no visible ghosting on orbit** (reset
> works). +~0.69 ms render, +~8 MB ping-pong memory (thickness+smooth_z default-on; +4 MB if
> foam_history on). **"Visibly calmer in motion" is unverifiable from stills — the #1 live-
> review item.** Doc migration deferred to the final consolidated pass.

> Hero-water series plan 7 of 7 — the finishing pass. Inherits the *Cross-cutting
> decisions* in [`v1.12.0-hero-water-refraction.md`](v1.12.0-hero-water-refraction.md).
> Lands last because it stabilizes the noisy outputs of v1.12–v1.17.

## Mission

Kill the shimmer/flicker that accumulates across the hero stack — thickness, normals,
caustics, foam, micro-normals — with per-target temporal history blending and a clean
reset on camera motion. Done when the dam-break capture in motion is visibly calmer than
the same scene without temporal, with no smearing or ghosting on camera moves.

## Reality check — this is more work than the source plan implies

The ChatGPT plan lists temporal as an "hours 6–7" polish item using "conservative
reprojection if camera matrices and velocity are available." **They are not.** The app
bakes the model matrix into `view_proj` and rotates billboards to compensate
([`rendering.md`](../architecture/rendering.md)); there is **no motion-vector / history
infrastructure** today. Real reprojection (motion vectors, neighborhood clamping,
disocclusion) is its own project. This plan deliberately scopes the **achievable**
version: exponential history blend per target + hard reset on camera move. Reprojection
is a future follow-up, not this plan.

## Scope

**In:**
- Ping-pong history targets for the buffers worth stabilizing: smoothed thickness,
  surface normal, caustics, foam target.
- Per-target exponential blend: `out = mix(current, history, history_alpha)`.
- A camera-delta uniform; **hard reset** (drop history, alpha→0 for one frame) when
  camera motion exceeds `camera_motion_reset_threshold`.
- Simple validity rejects: depth-difference and normal-difference thresholds to avoid
  blending across surfaces that moved.
- Wire the existing stopgap blends in v1.16 (caustics) under this unified control.
- Debug: per-target history-age / reject visualization.

**Out:** true motion-vector reprojection, neighborhood variance clamping, TAA-style
jitter (`jitter_enabled` stays off). Those need the motion-vector infra this plan doesn't
build.

## Approach

Touches each stabilized pass to add a history read/write; otherwise self-contained.

1. Camera-delta uniform + reset logic.
2. Ping-pong history targets (new GPU memory — account for it).
3. Per-target blend + validity rejects, starting with thickness/normals.
4. Fold in caustics + foam history.
5. Capture Scene 3 (dam break) in motion + a deliberate camera move; verify calmer, no
   ghosting; profiler.

## Watch-outs

- **Memory cost is real:** every stabilized target doubles (ping-pong). At full-res
  thickness/normals plus half-res caustics this is the series' biggest memory add after
  v1.14 — record it in `gpu-resources.md`.
- **Ghosting on camera moves** is the failure mode the reset threshold guards; tune it
  against an actual orbit capture, not a static frame.
- **Reset on camera move only** misses *content* motion (fast water with a static camera
  still flickers a little). Accept that for v1; document it as the known limitation that a
  future reprojection follow-up would address.
- Validity rejects need the depth/normal targets to be readable at history time — order
  the passes so history blend happens after the smoothed depth/normal exist.

## Concrete implementation (orchestrator-refined, 2026-06-09)

Code-grounded against the committed v1.12–v1.17 baseline. Source-of-truth precedence:
runtime Rust/WGSL > this section > the draft above > architecture docs.

### Reality corrections to the draft

Two items in the draft above are wrong against the code and are corrected here:

1. **There is no "surface normal" target to ping-pong, and no "foam target" either.**
   - The water normal is finite-differenced from `smooth_z` on the fly in
     `composite.wgsl::water_normal` (`textureLoad(smoothed_z_tex,…)` neighbors → cross
     gradient). It is NOT a stored target. **Stabilizing `smooth_z` IS stabilizing the
     normal** — there is nothing else to blend for normals.
   - The v1.13 foam/diffuse system (`gpu/diffuse.rs`) is a *geometry* particle system:
     instanced camera-facing billboards drawn straight to the swapchain in the "diffuse
     particle pass", premultiplied-alpha over the composite. There is **no screen-space
     foam render target**. The only screen-space "foam-ish" signal is `whitewater_view`
     (the R16 speed-weighted whitewater target written alongside thickness in the
     "water thickness pass"), which composite.wgsl reads for foam/roughness.
   - So "foam_history" in this plan means **`whitewater_view` history**, not the diffuse
     particles (which temporal blending cannot touch — they're alpha-blended geometry).

2. **The real ping-pong targets are the three full-res R16 screen-space targets**
   `thickness_view`, `smooth_z_view`, `whitewater_view`, plus the **already-ping-ponged**
   half-res caustic target. No marching-cubes / MC target exists (v1.14 abandoned).

### Stabilizable targets (real names)

| Logical | Target (mod.rs) | Res / format | Feeds | History add |
| --- | --- | --- | --- | --- |
| thickness | `thickness_view` | full-res R16Float | composite absorption/refraction bend | +1 R16 |
| front-depth (→ normals) | `smooth_z_view` | full-res R16Float | `water_normal` finite-diff, curvature, refraction | +1 R16 |
| foam | `whitewater_view` | full-res R16Float | composite foam/roughness | +1 R16 |
| caustics | `caustic_ping/pong_view` | half-res R16Float | scene_color additive composite | **0** (already ping-pong) |

`thickness_view` and `smooth_z_view` are the two that actually matter for the hero look
(`smooth_z` drives both the normal and the curvature/roughness). `whitewater_view` is
optional/cheaper-value — keep it behind its own `foam_history` flag.

### Approach: a single reusable temporal-blend pass over R16 targets

Add a new `gpu/temporal.rs` `TemporalSystem` owning one **history texture per stabilized
target** (not full ping-pong of both halves — see memory note) plus a small uniform. The
blend is a fragment pass: `out = mix(current, history, alpha)` written into the live
target's slot via a ping/pong, with per-target validity reject. Concretely the cleanest
wiring that avoids read-write hazards is **one extra R16 "stabilized" texture per target**
that the pass writes (sampling `current` + `history`), then **history := stabilized** is
achieved by making the *stabilized* texture also serve as next frame's history (true
ping-pong: two textures per stabilized target, roles swap each frame, exactly like
caustics). Downstream consumers (composite, caustics-gen) bind the **stabilized** view
instead of the raw target.

This means: keep the raw `thickness_view` / `smooth_z_view` / `whitewater_view` as the
pass *outputs* (current), and add a `*_stab_a` / `*_stab_b` ping-pong pair per enabled
target. The blend pass reads (raw current, prev stabilized) → writes new stabilized;
composite/caustics read the new stabilized.

### Camera-delta source + reset evaluation

- `view_proj` reaches `GpuContext::render` already **model-baked** (`lib.rs:411`
  `self.camera.view_proj(aspect) * model`). It is NOT usable for a clean camera delta
  because the box `model` is folded in. The clean, model-free camera signal is
  **`eye_to_world`**, built in `lib.rs:424` from `billboard_basis()` + `view_dir()` — it
  is camera-only (no box orientation). Pass the previous frame's `eye_to_world` rotation
  and diff it: `motion = acos(clamp((trace(R_prevᵀ·R_cur) - 1)/2, -1, 1))` (rotation
  angle between frames), optionally plus an eye-position delta. The eye position is
  recoverable in `lib.rs` from `camera.eye()` — simplest is to compute the scalar
  camera-motion metric **on the CPU in `render()` / the App tick** and pass it as a
  uniform float, rather than reconstructing it on the GPU.
- Recommended: store `prev_eye_to_world: Mat4` (+ `prev_eye: Vec3`) in `GpuContext`;
  each frame compute `cam_motion = rotation_angle + position_delta_scaled`; compare to
  `hero.camera_motion_reset_threshold`. If exceeded → set the per-target blend `alpha`
  effective value to 0 for this frame (history dropped), exactly mirroring the caustics
  `history_valid=false` first-frame suppression already in `caustics.rs::draw_generate`.
- The threshold is evaluated **CPU-side in `GpuContext::render`** (cheap, no readback) and
  pushed into the temporal uniform as an `alpha_override` (0 on reset frame, else the
  configured `history_alpha`). This is the same mechanism caustics already uses via
  `TEMPORAL_ENABLED_OFFSET` write_buffer.

### Render-order placement (gpu/mod.rs Water arm)

Current Water order: scene prepass → **thickness pass** (writes thickness/nearest_z/
whitewater) → smooth-x (nearest_z→smooth_z_ping) → smooth-y (ping→smooth_z) → caustics
gen+composite → water composite → diffuse particles.

Insert the temporal blend passes **after smooth-y completes** (so `smooth_z` exists) and
**after the thickness pass** (so thickness/whitewater exist), i.e. immediately *before*
the caustics gen pass and the water composite:

1. thickness pass (writes raw `thickness_view`, `nearest_z_view`, `whitewater_view`)
2. smooth-x → `smooth_z_ping`
3. smooth-y → `smooth_z_view` (raw current front-depth)
4. **NEW temporal-blend passes** (only if `hero.temporal_enabled`):
   - thickness: blend raw `thickness_view` + prev → `thickness_stab`
   - front-depth: blend raw `smooth_z_view` + prev → `smooth_z_stab`
     (depth-reject + normal-reject sampled from raw + prev smooth_z here)
   - foam: blend raw `whitewater_view` + prev → `whitewater_stab` (if `foam_history`)
5. caustics gen now reads **`smooth_z_stab` + `thickness_stab`** (validity rejects need
   the stabilized depth readable — satisfied, they exist from step 4)
6. caustics composite (unchanged target: scene_color additive)
7. water composite reads **stabilized** thickness/smooth_z/whitewater views
8. diffuse particles (unchanged)

When `temporal_enabled` is false, downstream binds fall back to the raw views (either via
a stored "active view" the composite/caustics `set_views` selects, or by always running
the blend with `alpha=0` which is a no-op copy — the copy-every-frame option is simpler
and keeps bind groups static, at the cost of one extra full-res pass when disabled; prefer
the **bind-swap** so disabled = zero added cost, matching the caustics "skip pass when
disabled" convention).

Validity rejects (depth-diff `depth_reject_threshold`, normal-diff `normal_reject_threshold`)
are computed inside the front-depth blend shader: compare current vs history `smooth_z`
(depth) and the finite-diff normal from each; if either exceeds threshold, drop to
`alpha=0` for that pixel (use current only). This needs both current and history smooth_z
readable — both are bound, so ordering is satisfied.

### Unifying the v1.16 caustics stopgap

Today caustics owns its own temporal blend inside `caustics_generate.wgsl` driven by
`caustics2.z = caustics_temporal_enabled` and `caustics2.w = caustics_temporal_alpha`
(read in `make_gen_uniform`, suppressed on first frame via `history_valid`). Re-home it:

- Keep the caustics generation ping/pong and its in-shader blend (it's already the right
  shape and half-res), but **drive its `temporal_enabled`/`history_alpha` from the unified
  `render.hero.temporal.*` values** instead of the separate `caustics.temporal_*` keys.
  In `make_gen_uniform`, source `caustics2.z` from `hero.temporal_enabled &&
  hero.caustic_history` and `caustics2.w` from `hero.history_alpha`.
- Apply the **same camera-motion reset** to caustics: when `cam_motion` exceeds the
  threshold, suppress the caustics in-shader blend that frame (reuse the existing
  `TEMPORAL_ENABLED_OFFSET` write-zero path in `draw_generate`, now also triggered by the
  reset, not only first-frame `history_valid`).
- Deprecate `render.hero.caustics.temporal_enabled` / `caustics.temporal_alpha`: either
  remove them or make them read-through aliases of the unified keys. Recommended: **remove
  the two caustics-specific settings** and the `caustics_temporal_enabled` /
  `caustics_temporal_alpha` `HeroParams` fields, replacing reads with the unified
  `temporal_enabled`/`history_alpha`. (Their tooltips already announce the v1.18 rename.)
  This leaves ONE temporal system, as required.

### Memory cost

Baseline (1280×800): each full-res R16 target ≈ 2 MB; `scene_color` Rgba16Float ≈ 4 MB.
This plan adds **true ping-pong (two textures) per enabled full-res target**:

- thickness history ping/pong: 2 × ~2 MB = **~4 MB**
- smooth_z (front-depth/normals) history ping/pong: 2 × ~2 MB = **~4 MB**
- whitewater (foam) history ping/pong: 2 × ~2 MB = **~4 MB** (only if `foam_history`)
- caustics: **0** new (already ping-pong half-res)

So **≈ 8 MB** with the two default-on full-res targets (thickness + smooth_z), **≈ 12 MB**
if foam_history is also enabled, at 1280×800 — scaling with resolution. This is the
largest memory add of the series after the abandoned v1.14. (A cheaper single-history-
texture variant exists — keep raw target as "current" and one history texture, blending
into a third — but two-texture ping-pong is the simplest hazard-free shape matching the
existing caustics pattern; record whichever ships in `gpu-resources.md`.)
Record alongside the existing "five R16 water targets ≈ 10 MB" line.

### Risks

- **Read-write hazard:** a stabilized texture must not be both sampled (history) and the
  render-pass color attachment in the same pass. The two-texture ping-pong avoids this
  (read prev, write new). Verify bind groups never alias the active attachment — this is
  the exact trap caustics' additive-blend comment calls out.
- **Ghosting on orbit:** the camera-motion reset is the only guard; tune
  `camera_motion_reset_threshold` against a real `DRAG=1` orbit capture, not a static
  frame (per exit gate). Content-motion flicker (fast water, static camera) is NOT
  addressed — documented limitation.
- **Reset metric correctness:** `view_proj` is model-baked; using it for the delta would
  trigger spurious resets on box rotation/translation. Must use `eye_to_world`
  (camera-only). Getting this wrong = either constant reset (no smoothing) or no reset
  (ghosting on orbit).
- **Caustics double-blend:** if the unified pass also blended `smooth_z`/`thickness` that
  feed caustics gen AND caustics keeps its own in-shader blend, the caustic map is blended
  twice. Acceptable (different signals: stabilized inputs vs stabilized output) but tune
  alphas; document. Do not also add a separate full-res caustic ping-pong — reuse the
  existing half-res one.
- **R16 precision:** depth-reject thresholds operate on R16Float eye distance with a
  65504 "no geometry" sentinel; the blend/reject must preserve the sentinel (don't blend
  a valid pixel with a sentinel history pixel — treat sentinel history as invalid →
  alpha 0), mirroring the `>= 60000.0` guards already in composite.wgsl and the smoothing
  clear color.
- **Resize/Outdated parity:** mod.rs rebuilds targets in three places (`resize`, the
  `Outdated|Lost` arm of `render`, and indirectly `recreate_fluid` does NOT rebuild these
  swapchain targets). The temporal history textures + `history_valid` flag must be reset
  in `resize` and the `Outdated` arm (NOT `recreate_fluid`, which leaves swapchain targets
  alone), and `history_valid=false` after each so the first post-resize frame doesn't
  ghost — exactly the caustics `set_views` pattern.
- **Caustics settings removal is a breaking settings-id change:** removing
  `render.hero.caustics.temporal_*` must update panels.js and any persisted-settings
  migration; the tooltips already pre-announce the rename, easing this.
